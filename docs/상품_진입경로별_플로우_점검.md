# 상품 진입경로별 플로우 점검 (클라이언트 관점)

상품 보기 → 선택 → 예약 → 결제까지, **메인 / 검색 / 서브메뉴 / 타임딜 / 프로모션 / 위시** 등 진입 경로별로 정리한 점검 결과입니다.

---

## 1. 진입 경로 요약

| 진입 경로 | URL/트리거 | 다음 화면 | 비고 |
|-----------|------------|-----------|------|
| **메인** | `/` (홈) → 최근/찜 상품 클릭 | `addModal('/product/{id}')` | 달력으로 이동 |
| **검색** | GET/POST `/product/search` → 결과에서 클릭 | `addModal('/product/{id}')` | 동일 |
| **서브메뉴** | `/menu/{code}` (101, 201, 601 등) → 상품 클릭 | `addModal('/product/{id}')` | 동일 |
| **타임딜** | `GET /timedeal` → 상품 클릭 | `addModal('/product/{id}')` | productResult 동일 템플릿 |
| **프로모션** | `GET /promotion/{id}` → 상품 클릭 | `addModal('/product/{id}')` | 동일 |
| **위시(찜/최근)** | 마이페이지 위시/최근목록 → 클릭 | `addModal('/product/{id}')` | 동일 |

**공통점**: 모든 경로에서 상품 클릭 시 **`/product/{id}`(달력 페이지)** 로만 이동하며, `departDate` 등은 넘기지 않음.

---

## 2. 플로우 단계별 정리

### 2.1 상품 목록 → 달력 (1단계)

- **진입**: 메인/검색/서브/타임딜/프로모션/위시에서 상품 클릭.
- **요청**: `GET /product/{id}` → `ProductController.getProduct()`.
- **화면**: `productCalander` (출발일 선택 달력 + 날짜별 최저가 표시).
- **데이터**: `calendarProductsJson`, `listProductsByDate` (날짜별 항공 조합) 전달.
- **로직**: `findMultipleProductsForDate` + `hasHoldWithPoolSpace`로 노출 조건 적용, 동일 항공사·좌석등급 필터 적용됨.

**점검 결과**: 진입 경로와 관계없이 항상 달력으로만 들어오므로, 한 경로만 확인하면 됨. 문제 없음.

---

### 2.2 달력 → 날짜 선택 → 하단 리스트 (2단계)

- **동작**: 달력에서 날짜 클릭(`dateClick` / `eventClick`) → `showProductsForDate(dateStr)`.
- **데이터**: `listProductsByDate[dateKey]` (해당 날짜의 `ProductCalanderDto[]`).
- **표시**: 출발·귀국 시간, 좌석등급, 가격, 잔여석 등.

**수정 반영**:
- `p.departDateTime` 없으면 리스트 항목 스킵.
- `departDate` 파싱 실패(Invalid Date) 시 스킵.
- `returnDateTime` null 시 `returnDateStr = '-'` 로 표시.
- 상세 링크 생성 시 `departDateTime`/`returnDateTime`을 문자열로 안전하게 변환 후 `split('T')` 사용 (객체 직렬화 대비).

---

### 2.3 리스트 항목 클릭 → 상세 (3단계)

- **동작**: 해당 날짜 리스트에서 한 항목(한 항공 조합) 클릭.
- **요청**:  
  `GET /product/detail/{id}?departDate=...&departTime=...&returnTime=...&seatClassType=...`
- **화면**: `productDetail` (상세 + 예약하기 버튼).
- **백엔드**: `getProductDetail()` 에서 `departTime`/`returnTime`/`seatClassType` 로 동일 조합 매칭 후 `calcProduct` 세션 저장, 모델에 `departTime`/`returnTime`/`seatClassType` 도 넣어서 hidden으로 전달.

**점검 결과**:
- 달력 리스트에서만 상세로 들어오므로, **항상** `departDate` + (선택) `departTime`/`returnTime`/`seatClassType` 이 전달됨.
- 상세에서 예약하기 시 `reservation/save` 또는 모달로 넘기는 payload 에 `departTime`/`returnTime`/`seatClassType` 포함되도록 이미 수정됨 → 로그인 후 복구 시 동일 조합 유지.

---

### 2.4 상세 → 예약하기 (4단계)

- **비로그인**: `POST /product/reservation/save` 로 `productId`, `departDate`, `adult`/`youth`/`infant` + (있으면) `departTime`/`returnTime`/`seatClassType` 저장 → 로그인 유도.
- **로그인**: `CustomAuthenticationSuccessHandler` 에서 `pendingReservation` → `reservationData` 로 복사.
- **예약 모달**: `addModal('product/reservation', false, reservationData)` 또는 `POST /product/reservation` 호출 시, `reservationData`로 `calcedProduct` 복구 시 `departTime`/`returnTime`/`seatClassType` 있으면 `findMatchingCalcProduct` 로 동일 조합 복구.

**점검 결과**: 메인/검색/서브/타임딜/프로모션/위시 모두 **달력 → 날짜 → 리스트 → 상세** 를 거치므로, 예약 시점에는 항상 상세에서 넘긴 조합 정보가 있음. 복구 로직과 정합성 맞음.

---

### 2.5 예약 → 결제 → 탑승객 · 내역 (5단계)

- **예약 확정**: `POST /product/purchase` → `ensurePackageSeatHold`, `paymentDueDate`/`passengerInfoDeadline` 은 `cutoffDays` 기준 적용됨.
- **결제**: `GET/POST /product/purchase/{id}/payment` 에 본인 검증·Optional 처리 적용됨.
- **탑승객**: `GET/POST /purchase/product/{id}/passport`.
- **내역**: `/mypage/reservation` 에서 `findByUserId` 로 목록 노출.

**점검 결과**: 진입 경로와 무관하게 동일 플로우이며, 이미 적용된 수정 사항으로 일관성 유지됨.

## 3. 진입 경로별 체크리스트

| 경로 | 달력 진입 | 날짜 선택 | 리스트→상세 (조합 파라미터) | 예약/결제 |
|------|-----------|-----------|-----------------------------|-----------|
| 메인 | ✅ `/product/{id}` | ✅ 동일 | ✅ departTime/returnTime/seatClassType | ✅ |
| 검색 | ✅ 동일 | ✅ 동일 | ✅ 동일 | ✅ |
| 서브메뉴 | ✅ 동일 | ✅ 동일 | ✅ 동일 | ✅ |
| 타임딜 | ✅ 동일 | ✅ 동일 | ✅ 동일 | ✅ |
| 프로모션 | ✅ 동일 | ✅ 동일 | ✅ 동일 | ✅ |
| 위시 | ✅ 동일 | ✅ 동일 | ✅ 동일 | ✅ |

모든 경로가 **상품 클릭 → 달력 → (날짜 클릭) → 리스트 → (항목 클릭) → 상세 → 예약 → 결제** 로 통일되어 있음.

---

## 4. 적용한 수정 요약

1. **productCalander.html (달력 하단 리스트)**
   - `p.departDateTime` 없으면 항목 스킵.
   - `departDate` 파싱 실패(Invalid Date) 시 스킵.
   - `returnDateTime` null 일 때 `'-'` 로 표시, `returnDate` 사용 시 null 체크.
   - 상세 링크 생성 시 `departDateTime`/`returnDateTime` 문자열 변환 후 `split('T')` 사용해 런타임 에러 방지 및 `departTime`/`returnTime` 안정 전달.

2. **기존 적용 사항 (이전 리포트 반영)**
   - 상세에서 `departTime`/`returnTime`/`seatClassType` 모델 전달 및 hidden 입력.
   - 예약 시 payload 에 위 3개 포함 → `reservationData`/`pendingReservation` 복구 시 동일 조합 사용.
   - 결제 전 취소 시 `cancelPurchase` 호출, 결제/문의 본인 검증, `calcedProduct` null 방지, `cutoffDays` 기준 결제 기한 등.

---

## 5. 결론

- **메인 / 검색 / 서브메뉴 / 타임딜 / 프로모션 / 위시** 모두 동일하게 **상품 클릭 → 달력 → (날짜) → 리스트 → 상세 → 예약 → 결제** 를 따릅니다.
- 달력 리스트에서만 상세로 들어가므로, 상세·예약·결제 단계에서 “선택한 항공 조합”은 항상 `departDate` + `departTime`/`returnTime`/`seatClassType` 으로 식별 가능합니다.
- 이번에 달력 리스트의 null·파싱 안전성을 보강해, 어떤 진입 경로에서 와도 리스트 클릭 시 상세로 안정적으로 넘어가고, 예약 복구 시에도 동일 조합이 유지되도록 맞춰 두었습니다.
