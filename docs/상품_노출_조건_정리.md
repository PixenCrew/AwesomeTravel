# 상품 노출 정책 계획

## 전제 (유지하는 것)

- **비즈니스/계약 조건은 그대로 유지**: 항공사 일치, 한 상품=한 좌석등급, 투어 기간, cutoffDays. (departTimeType만 선택 필터로 완화.)
- **메인 / 타임딜 / 기획전**: "이용 가능한 상품만" 노출 (현재처럼 `calcedProduct != null`인 경우만 표시). 변경 없음.

## 변경 범위 요약

| 구역 | 현재 | 변경 후 |
|------|------|--------|
| 검색 결과 | 이용 가능한 상품만 표시 | 조건 안 맞아도 상품 노출 (대표 가격 없으면 "문의" 등 처리) |
| 메뉴별 상품 목록 | 이용 가능한 상품만 + 없으면 에러 페이지 | 조건 안 맞아도 상품 노출, 없으면 빈 목록 또는 안내 |
| 상품 상세(달력) | `hasNoProducts` 시 "판매하지 않는 상품" 메시지 | "예약 가능한 일정이 없습니다" 등으로 문구만 변경 |

## 1. 검색 결과: 조건 안 맞아도 상품 노출

**파일:** [ProductController.java](src/main/java/renewal/awesome_travel/product/controller/ProductController.java) `postProductSearch` (약 127~178라인)

**현재:** `calcedProduct != null`일 때만 `calcedResult.add(calcedProduct)` 하므로, 100일 구간 내에 한 번도 가격 계산이 성공하지 않으면 검색 결과에 안 나옴.

**변경:**

- 검색된 `Product`에 대해 루프를 돌 때, `calcedProduct == null`이어도 **상품은 결과에 포함**하도록 수정.
- 이때 대표 가격/출발일이 없으므로:
- **방안 A**: 원본 `product`를 그대로 add하고, `finalPriceAdult`/`price`는 null 또는 0으로 두고, 템플릿에서 "가격 문의" 또는 "일정 선택 시 가격 확인" 등으로 표시.
- **방안 B**: `calcedProduct`가 null이면 "이용 불가" 플래그만 가진 DTO/Product를 add하고, 목록 UI에서 해당 항목은 가격 대신 "문의" 표시 후 클릭 시 상세로 이동.

**템플릿:** [productResult.html](src/main/resources/templates/fragments/product/productResult.html)

- 목록에서 `price`/`finalPriceAdult`가 없거나 플래그가 "이용 불가"일 때 문구/스타일 처리 (예: "가격 문의", "일정 선택 시 확인").

## 2. 메뉴별 상품 목록: 조건 안 맞아도 상품 노출

**파일:** [MainController.java](src/main/java/renewal/awesome_travel/MainController.java) `getMenuByCode` (약 404~445라인)

**현재:** `calcedProduct != null`일 때만 `resulProducts.add(calcedProduct)`. `resulProducts.isEmpty()`면 `renderDataError`로 "현재 준비 중인 상품이거나 데이터가 아직 등록되지 않았습니다." 반환.

**변경:**

## 4. departTimeType: 선택 필터로 완화

**의도:** 출발 시간대를 필수 조건이 아니라 "선택 필터"로만 사용. 조회 시에는 하루 전체(00:00~23:59) 항공편을 가져오고, 필요 시 UI에서 사용자가 출발 시간대를 골라 필터링.

**파일:** [ProductServiceCommon.java](src/main/java/renewal/common/service/ProductServiceCommon.java) `calcSingleProduct` (약 204~210라인)

**현재:** Day 0일 때 `product.getDepartTimeType()`의 `getStartHour()`, `getEndHour()`로 시간대를 제한해 `findLowestPriceSeat` 호출 (예: MORNING이면 06:00~11:59만 조회).

**변경:**

- Day 0이어도 **항상** `startHour = 0`, `endHour = 23`으로 두어 **하루 전체** 구간으로 항공편 조회.
- 즉, `int startHour = 0;`, `int endHour = 23;`으로 통일하거나, 기존 `dtt.getStartHour()`/`getEndHour()` 분기 제거.

**참고:**

- 달력 페이지(ProductController.findMultipleProductsForDate)는 이미 00:00~23:59로 조회 중이므로 변경 없음.
- 이후 UI에서 "출발 시간대" 드롭다운(아침/오전/오후 등)을 두어 사용자가 선택 시 목록/달력 결과를 필터링하는 것은 선택 사항.

## 6. 패키지 결제 확정 시 출국/귀국 좌석 둘 다 차감

**의도:** 패키지는 같은 인원이 출국·귀국 모두 이용하므로, 결제 확정 시 **출국편 SeatClass**와 **귀국편 SeatClass**에 **동일 인원 수 N**을 각각 차감해야 함.

**현재:** `PurchaseProduct` 생성 시 `finalSeatClasses`에는 출국/귀국 각각 같은 adult, youth, infant로 저장됨. 하지만 **SeatClass.availableSeats**를 차감하는 로직이 없음 (결제 확정 시에도 미호출).

**추가할 로직:**

- **위치:** [ProductController.java](src/main/java/renewal/awesome_travel/product/controller/ProductController.java) `postPurchasePayForm` (결제 확정 처리, 약 2249~2284라인). 결제 상태를 PAID로 업데이트한 뒤, `PurchaseProduct.getFinalSeatClasses()`를 순회하며 각 `ConfirmedSeatClass`에 대응하는 SeatClass 엔티티를 조회(airId + classType)하고, **같은 N(adult + youth)**으로 `SeatClass.reserveSeats(N)` 호출.
- **효과:** 출국 1건당 N석 차감, 귀국 1건당 N석 차감 → 패키지 1건 예약 시 출국·귀국 모두 N석 차감되어 동기화됨.
- **취소 시:** 예약 취소 처리 시 해당 `PurchaseProduct`의 `finalSeatClasses`를 보고 각 SeatClass에 대해 `cancelSeats(N)`(또는 동일 인원 복원) 호출 로직도 필요.

**참고:** [SeatClass.java](src/main/java/renewal/common/entity/SeatClass.java)에 `reserveSeats(Long requiredPersons)`, `cancelSeats(Long canceledPersons)` 이미 정의됨. [PurchaseBase.ConfirmedSeatClass](src/main/java/renewal/common/entity/PurchaseBase.java)는 airId, classType, seatCountAdult, seatCountYouth 등 보유.

## 7. 패키지 좌석 미리 확보 및 반납 (가능 인원·최소 인원)

**의도:** 패키지는 "가능 인원 20명·최소 인원 12명"처럼 정해진 경우, 항공 좌석을 **미리 패키지용으로 빼두지 않으면** 일반 항공 판매에 선점당해 패키지에 쓸 좌석이 없어질 수 있음. 또 최소 인원 미달 시 패키지 취소·반납, 최소만 채우고 남은 좌석 반납이 필요함.

**추가할 정책·로직:**

- **미리 빼두기(홀드):** 특정 상품·출발일·항공 조합에 대해 **가능 인원(예: 20명)**만큼 출국/귀국 SeatClass에서 "패키지 홀드"로 미리 차감(또는 홀드 전용 필드/테이블로 관리). 시점은 상품·출발일 오픈 시 또는 첫 예약 접수 시 등 정책에 따라 결정.
- **최소 인원 미달 시:** 해당 출발일 패키지로 잡아둔 좌석 **전부** 출국·귀국 SeatClass에 반납(cancelSeats). 패키지 취소 처리(환불 등)와 함께 수행.
- **최소 인원은 채웠지만 가능 인원 미달 시(예: 12명만 예약):** 실제 예약 인원(12명)만 확정 유지하고, **남은 8석은 "패키지 홀드"에서 해제**해 SeatClass에 반납. (반납 시점: 출발일 N일 전 마감 시점 등 정책 결정.)
- **구현 시 고려:** "홀드" vs "확정(결제 완료)" 상태 구분, 반납 시점(패키지 취소 시점·출발 X일 전 마감), Tour/Product의 maxCapacity·minCapacity(또는 동일 의미 필드) 활용.

**추가할 것:**

- 패키지용 좌석 홀드/해제를 담당하는 서비스 또는 도메인 로직.
- 결제 확정(6번)과 연동: 홀드된 좌석 중 N명분을 "확정"으로 전환하고, 필요 시 남은 홀드 분은 마감 시점에 반납.
- 패키지 취소(최소 미달 포함) 시 해당 출발일 홀드/확정 분 전부 SeatClass에 반납.

## 8. 달력 화면: 조건에 맞는 여러 상품 표시 및 좌석등급 필터

**현재 구조:**

- **calendarProducts:** 날짜별 대표 상품 리스트(ProductCalanderDto). 달력 셀 하이라이트·이벤트용.
- **listProductsByDate:** `Map<날짜(YYYY-MM-DD), List<ProductCalanderDto>>`. 날짜 클릭 시 하단에 보여줄 "이코노미/비즈니스 등 여러 옵션" 목록.
- **필터:** `seatClassFilter`(이코노미/비즈니스 등)는 `applyListFilters()`에서 `currentListProducts`를 `p.seatClassType === 선택값`으로 필터링. `ProductCalanderDto.seatClassType`은 한글("이코노미", "비즈니스" 등)으로 설정됨.

**의심 원인 (필터/여러 상품이 안 보일 수 있는 경우):**

1. **서버 → 클라이언트 직렬화:** `model.addAttribute("calendarProducts", ...)`, `model.addAttribute("listProductsByDate", ...)` 후 템플릿에서 `/*[[${calendarProducts}]]*/`, `/*[[${listProductsByDate}]]*/`로 넘기면, Thymeleaf는 복잡 객체를 기본적으로 JSON으로 직렬화하지 않음. List/Map이 toString() 형태로 나가면 브라우저에서 유효한 JSON이 아니라 **데이터가 비었거나 깨져서** 달력/하단 리스트가 비고, 필터를 걸어도 반응이 없을 수 있음.
2. **seatClassType이 null:** `ProductCalanderDto`는 모든 AIR Location에 동일한 SeatClassType이 있을 때만 `seatClassType`을 설정함. 일부 Location에 SeatClass가 없거나, enum이 switch에 없으면 null. 필터에서 "이코노미" 선택 시 `p.seatClassType !== "이코노미"`면 제외되므로, null인 항목은 **어떤 좌석등급을 선택해도 목록에 안 나올 수 있음**.

**추가/수정할 것:**

- **직렬화:** 컨트롤러에서 `calendarProducts`, `listProductsByDate`를 **JSON 문자열**로 변환(ObjectMapper 등)한 뒤 model에 문자열로 넣고, 템플릿에서는 해당 문자열을 그대로 출력해 스크립트에서 `JSON.parse(...)` 하도록 변경. 이렇게 해야 `seatClassType`, `departDateTime`, `status` 등이 그대로 프론트에 전달됨.
- **seatClassType 보강:** `ProductCalanderDto` 생성 시, AIR Location 중 **하나라도** SeatClass가 있으면 그 구간의 classType으로라도 `seatClassType`을 설정하도록 fallback 추가(모두 동일한 경우만이 아니라, 첫 번째 유효한 값이라도 넣기). 단, "한 상품 = 한 좌석등급" 정책이 유지되므로 기존 로직으로도 대부분 설정되나, 예외 경로에서 null이 나오지 않도록 확인.
- **날짜 키 일치:** `listProductsByDate` 키는 `targetDate.toString()` (YYYY-MM-DD). 클라이언트는 `dateClick`의 `info.dateStr` 또는 이벤트 날짜로 YYYY-MM-DD를 만들어 사용. 타임존으로 인해 날짜가 어긋나지 않도록, 클릭 시 사용하는 날짜 문자열이 서버 키와 동일한 형식인지 확인.

**정리:** "조건에 맞는 여러 상품 표시" 로직 자체(날짜별 listProductsByDate, 좌석등급별 Product 생성)는 올바르게 되어 있으나, **달력/리스트 데이터가 프론트로 유효한 JSON으로 전달되는지**, 그리고 **모든 DTO에 seatClassType이 채워지는지**를 점검·수정하면 필터와 여러 상품 표시가 정상 동작할 가능성이 높음.

## 구현 순서 제안

1. **상세 달력 문구 변경** (productCalander.html) — 리스크 적고 즉시 적용 가능.
2. **departTimeType 선택 필터 완화** — ProductServiceCommon에서 Day 0도 00:00~23:59로 조회하도록 수정.
3. **검색 결과** — ProductController에서 `calcedProduct == null`이어도 상품 추가 + productResult에서 가격 없음/문의 표시.
4. **메뉴별 목록** — MainController.getMenuByCode에서 동일하게 이용 불가 상품도 포함, 빈 목록/에러 처리 정리.
5. **패키지 결제 확정 시 출국/귀국 좌석 둘 다 차감** — postPurchasePayForm에서 finalSeatClasses 기준으로 각 SeatClass에 reserveSeats(N) 호출. (취소 시 cancelSeats 복원 로직도 추가.)
6. **패키지 좌석 미리 확보 및 반납** — 가능 인원만큼 홀드, 최소 인원 미달 시 전부 반납, 최소만 채운 경우 남은 좌석 반납. (홀드/확정 상태, 반납 시점 정책 정의 후 구현.)
7. **달력 화면 데이터 직렬화 및 좌석등급 필터** — calendarProducts/listProductsByDate를 JSON 문자열로 전달하도록 수정, ProductCalanderDto의 seatClassType null 방지, 날짜 키 형식 일치 확인.

## 주의사항

- 이용 불가 상품을 목록에 넣을 때 **동일 Product 엔티티**를 넣으면 `finalPriceAdult`/`price`가 null일 수 있음. 목록 템플릿에서 null/0 처리와 "가격 문의" 등 표시를 반드시 넣어야 함.
- 상세 페이지(`/product/{id}`)는 이미 조건 불일치 시 진입 가능하고 `hasNoProducts`로 달력만 비우고 있으므로, 문구 변경만으로 의도에 맞게 정리됨.
- 좌석 차감 시 **출국·귀국 모두 같은 N으로 차감**해야 패키지 정책과 일치함. 취소 시에도 두 구간 모두 동일 인원만큼 복원해야 함.
- 패키지 좌석 미리 확보(7번)는 6번과 함께 설계 시 "홀드 → 확정" 전환과 "남은 좌석 반납" 시점을 정한 뒤 구현하는 것이 좋음.
- 달력 화면(8번): 서버에서 List/Map을 그대로 model에 넣고 Thymeleaf로 스크립트에 넣으면 JSON이 아닌 형태로 나갈 수 있어, 이코노미/비즈니스 필터가 동작하지 않거나 여러 상품이 안 보일 수 있음. 반드시 JSON 문자열로 넘겨 파싱하도록 할 것.
