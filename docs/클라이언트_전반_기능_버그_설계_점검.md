# 클라이언트 프로젝트 전반 기능·버그·설계 점검

진입 경로·컨트롤러·템플릿·API를 기준으로 기능, 버그, 설계 이슈를 정리한 점검 문서입니다.

---

## 1. 보안 (Security)

### 1.1 심각: API에서 userId를 파라미터로 받는 경우

| 위치 | 문제 | 권장 |
|------|------|------|
| **NotificationApiController** (`/api/notifications`) | `GET ?userId=`, `PATCH /read?userId=` 로 **다른 사용자 ID**를 넘기면 해당 사용자의 알림 조회·읽음 처리 가능 | `@AuthenticationPrincipal`로 현재 로그인 사용자만 사용. `userId` 파라미터 제거 후 서버에서 principal 기반으로 처리 |
| **InquiryUserController** (`/api/inquiry`) | `GET ?userId=`, `GET /{id}?userId=`, `GET /search?userId=` 로 **다른 사용자 문의** 조회 가능 | 동일하게 principal 기반으로만 조회. `userId` 파라미터 제거 |

→ **타 사용자 데이터 열람/변경**이 가능한 구조이므로 우선 수정 권장.

### 1.2 인증·본인 확인 부족

| 위치 | 문제 | 권장 |
|------|------|------|
| **PurchaseController** `/purchase/{type}/{id}/passport` | 로그인·본인 여부 검사 없음. **예약 ID만 알면 누구나** 탑승객 폼 조회/수정 가능 | 로그인 필수 + 해당 예약의 `user.id`와 현재 사용자 일치 여부 검사 |
| **MyPageController** `getMateFragment`, `getMateInfo`, `deleteMate` | `PassportAccessConsent` id만으로 조회/삭제. **본인 소유 메이트인지 확인 없음** | `matePassport.getUser().getId().equals(currentUser.getId())` 검사 추가. 불일치 시 403/404 반환 |
| **SecurityConfig** | `csrf().disable()`, `anyRequest().permitAll()` 로 모든 경로 허용. 인증 필요 경로는 컨트롤러 내부에서만 검사 | (선택) 중요 API·폼은 `authorizeHttpRequests`로 로그인 필수 지정 고려 |

### 1.3 기타

- **GlobalControllerAdvice** `currentUser(principal)`: `principal.getUser()` null 체크 추가 **(반영됨)**.

## 2. 버그 (Bugs)

### 2.1 Optional / null 처리

| 위치 | 문제 | 권장 |
|------|------|------|
| **MyPageController** `mypageInquiryFragment` | `principal` null 미검사. 비로그인 시 `principal.getUser()` 에서 NPE | `if (principal == null \|\| principal.getUser() == null) return "fragments/login";` 등 처리 |
| **MyPageController** `mypageSettingFragment`, `mypageAccountSettingFragment`, `myPassportFragment`, `myPassportPost` | `userRepo.findByEmail(principal.getName()).get()` — 사용자 없으면 NoSuchElementException | `.orElseThrow()` 또는 null 체크 후 로그인 유도 |
| **MainController** `main`, `wish` | `userRepo.findByEmail(principal.getName()).get()` — 동일 | `.orElseThrow()` 또는 Optional 처리 |
| **AirController** `getPurchaseDetail` (결제 상세) | `userRepo.findByEmail(principal.getName()).get()` — 동일 | `.orElse(null)` 후 null이면 로그인 페이지 등 처리 |
| **PurchaseController** `getPassportForm` | `purchaseAirRepo.findById(id).get()`, `purchaseProductRepo.findById(id).get()` — 잘못된 id 시 예외 | `.orElse(null)` 후 없으면 에러 페이지/메시지 반환 |
| **RegisterController** `verifyNumberCheck` | `session.getAttribute("tempUser")` 가 null 이면 (세션 만료) `tempUser.getNumber()` 에서 NPE | `if (tempUser == null) return ResponseEntity.badRequest().body(...);` |
| **InquiryController** `getInquiryDetail` | 비공개 문의에서 `inquiry.getUser()` 가 null 이면 `inquiry.getUser().getId()` NPE | `inquiry.getUser() != null` 검사 후 비교 |

### 2.2 에러 뷰

| 위치 | 문제 | 권장 |
|------|------|------|
| 여러 컨트롤러 | `return "error/error";` 사용 | `templates/` 아래에 `error/error.html` 가 없음 (있음: `fragments/error/dataUnavailable.html`, `static/error/error.html`). 뷰 리졸버 설정에 따라 404/500 가능. `fragments/error/...` 또는 실제 존재하는 error 템플릿으로 통일 필요 |
| **PurchaseController** | `return "error";` | 동일. 실제 템플릿 경로와 맞는지 확인 |

### 2.3 기타

- **ReviewController** (댓글 등록/수정/삭제, 내 댓글): `userDetails.getUser()` 전에 `userDetails == null` 검사 없음. 비로그인 요청 시 NPE 가능 → 401 반환 처리 권장.
- **withdrawalFragment**: `principal.getName()` 호출 전에 principal null 검사 없음 (다른 메서드는 있는 경우 있음). 일관되게 principal null 체크 권장.

## 3. 설계·일관성 (Design)

### 3.1 인증 처리 방식 혼재

- **Principal** vs **@AuthenticationPrincipal CustomUserDetails** 혼용 (같은 마이페이지/상품에서도 혼재).
- 사용자 조회: `principal.getName()` → `userRepo.findByEmail(...).get()` vs `principal.getUser()`.
- 권장: 마이페이지·결제·문의 등 인증 필요 구간은 `CustomUserDetails` + null 체크로 통일하고, `principal.getUser()` 가 있으면 DB 재조회 최소화.

### 3.2 응답 형식

- API 성공/실패: `Map.of("success", true/false, "message", ...)` 와 `ResponseEntity.status(401).build()` 등 혼재.
- 권장: REST API는 HTTP 상태 코드 + 일관된 body 형식(예: `{ "success", "message" }`)으로 통일.

### 3.3 로깅·디버그 출력

- **RegisterController**, **MyPageController**: 휴대폰 인증 번호 등 `System.out.println` [TEST] 출력.
- **MainController**: 타임딜/기획전 디버그용 `System.err.println`.
- **UserService**, **SearchController**: `System.out` 사용.
- 권장: Logger로 교체, 운영 환경에서는 로그 레벨로 제어.

### 3.4 프론트엔드

- **layout.js** 단일 진입점, 여러 프래그먼트에서 인라인 스크립트 사용 — 버전/중복 주의 (예: FullCalendar 6.1.19 vs 6.1.11 언급된 부분).
- 에러 시 사용자 메시지: 일부는 `model.addAttribute("error", "로그인이 필요합니다.")` 등으로 전달, 일부는 뷰 이름만 반환. 메시지 노출 방식 통일 권장.

## 4. 기능별 요약

| 영역 | 기능 | 상태 | 비고 |
|------|------|------|------|
| 메인/홈 | `/`, 위시, 서브메뉴, 타임딜, 기획전 | 동작 | Optional.get() 버그 가능 |
| 로그인/회원가입 | 로그인, OAuth2, 회원가입(약관→폼→휴대폰→이메일 인증) | 동작 | register/phone/check 세션 null 시 NPE |
| 계정 복구 | ID/비밀번호 찾기 | 동작 | - |
| 상품 | 검색→결과→상세→달력→예약→결제→취소 | 동작 | 기존 리포트 참고(결제 전 취소, 조합 복구 등 반영됨) |
| 항공 | 검색→상세→예약→결제→취소 | 동작 | getPurchaseDetail 등 Optional 처리 보강 필요 |
| 예약/결제 | 상품·항공 예약, 결제, 탑승객 폼 | 동작 | 탑승객 폼 본인 검증 없음(보안) |
| 마이페이지 | 예약내역, 문의, 설정, 프로필, 여권, 메이트, 회원정보, 비밀번호, 탈퇴, 등급, 리뷰, 알림 | 동작 | inquiry 프래그먼트 NPE, 메이트 소유자 검증 없음, Optional.get() 다수 |
| 문의 | 문의 등록/목록/상세, API | 동작 | API에서 userId 파라미터로 타인 문의 조회 가능(보안) |
| 알림 | 알림 센터, 읽음 처리, API | 동작 | API에서 userId로 타인 알림 조회/수정 가능(보안) |
| 리뷰 | 댓글 CRUD, 신고, API | 동작 | userDetails null 시 NPE 가능 |
| 공지/FAQ/약관 | 목록·상세, API | 동작 | - |

## 5. 수정 우선순위 제안

| 우선순위 | 구분 | 항목 |
|----------|------|------|
| 1 | 보안 | 알림/문의 API에서 userId 파라미터 제거, principal 기반으로만 조회·수정 **(반영됨: NotificationApiController, InquiryUserController, notificationCenter.html)** |
| 2 | 보안 | 탑승객 폼(/purchase/{type}/{id}/passport) 로그인 + 예약 본인 확인 |
| 3 | 보안 | 메이트 조회/삭제 시 소유자 확인 **(반영됨: getMateFragment, getMateInfo, deleteMate)** |
| 4 | 버그 | MyPage /inquiry principal null 체크, RegisterController tempUser null 체크 **(반영됨)** |
| 5 | 버그 | Optional.get() → orElseThrow 또는 null 처리 (Main, MyPage, Air, Purchase 등) **(반영됨)** |
| 6 | 버그 | error/error 뷰 경로 확정 및 템플릿 생성 또는 fragments/error 로 통일 **(반영됨: 전부 fragments/error/dataUnavailable)** |
| 7 | 설계 | System.out/err → Logger, 인증 처리 방식·응답 형식 통일 **(반영됨: MainController, RegisterController, MyPageController, UserService, SearchController)** |
