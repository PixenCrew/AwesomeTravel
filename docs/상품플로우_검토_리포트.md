## 2. 오류 및 수정 권장 사항

### 2.1 결제 전 취소 시 홀드/대기열 미반영 (버그)

**위치**: `ProductController.java`  
`@PostMapping("/purchase/{id}/cancel")` — 결제 없이 취소하는 분기(2434~2439라인)

**현재 동작**:  
`payment == null`이면 `PurchaseProduct` 상태만 `CANCELLED`로 바꾸고 저장함.

**문제**:  
`productServiceCommon.cancelPurchase(id)`를 호출하지 않아서:

- 해당 예약이 사용 중이던 **PackageSeatHold.allocated**가 줄지 않음.
- **대기 예약 → 확정 전환** 로직이 돌지 않음 (cancelPurchase 내부 처리).

**권장 수정**:  
결제 전 취소도 공통 플로우를 타도록 변경.

```java
// 결제 전 예약 취소: 예약 상태만 변경
} else {
    productServiceCommon.cancelPurchase(id);  // 추가: 홀드 감소 + 대기열 처리
    return ResponseEntity.ok(Map.of("success", true, "message", "예약이 취소되었습니다."));
}
```

기존처럼 `cancelPurchase` 없이 상태만 바꾸면, 홀드 풀과 대기 예약 수가 실제와 어긋납니다.

### 2.2 결제 기한이 상품별 마감일과 불일치 (설계)

**위치**: `ProductController.java`  
`@PostMapping("/purchase")` — 2222~2223라인

**현재**:  
`paymentDueDate` / `passengerInfoDeadline`을 **출발 5일 전**으로 고정.

```java
purchaseProduct.setPaymentDueDate(calcedProduct.getDepartDateTime().minusDays(5));
purchaseProduct.setPassengerInfoDeadline(calcedProduct.getDepartDateTime().minusDays(5));
```

**문제**:  
상품별 마감일은 이미 `Product.cutoffDays`로 정의되어 있고, 스케줄러도 이 값으로 동작하는데, 결제/탑승객 기한만 5일로 고정되어 있음.

**권장**:  
상품 마감일과 맞추기.

```java
int cutoff = (calcedProduct.getCutoffDays() != null) ? calcedProduct.getCutoffDays().intValue() : 5;
purchaseProduct.setPaymentDueDate(calcedProduct.getDepartDateTime().minusDays(cutoff));
purchaseProduct.setPassengerInfoDeadline(calcedProduct.getDepartDateTime().minusDays(cutoff));
```

### 2.3 예약 단계에서 calcedProduct null 시 NPE (버그)

**위치**: `ProductController.java`  
`@PostMapping("/reservation")` — showPurchasePage (2061~2132)

**상황**:  
`calcProduct`가 세션에 없고, `reservationData`도 없거나 `productId`/`departDate`가 없으면 `calcedProduct`가 null인 채로 아래 코드가 실행됨.

```java
if (calcedProduct.getAvailableSeats() < adult + youth) {  // NPE
```

**권장**:  
복구 로직 직후에 null 체크 추가.

```java
if (calcedProduct == null) {
    model.addAttribute("error", "상품 정보를 확인할 수 없습니다. 상품 상세에서 다시 선택해 주세요.");
    return "error/error";
}
```

### 2.4 결제 요청 시 calcedProduct null 가능 (버그)

**위치**: `ProductController.java`  
`@PostMapping("/purchase")` — purchaseModal (2134~)

**상황**:  
`session.getAttribute("calcProduct")`가 null이면(세션 만료/다른 탭 등) `calcedProduct`가 null인데, 곧바로 `calcedProduct.getFinalPriceAdult()` 등으로 사용하면 NPE.

**권장**:  
맨 앞에서 null 체크 후 에러 응답 또는 에러 뷰 반환.

```java
Product calcedProduct = (Product) session.getAttribute("calcProduct");
if (calcedProduct == null) {
    model.addAttribute("error", "세션이 만료되었거나 상품 정보를 찾을 수 없습니다. 상품 상세에서 다시 선택해 주세요.");
    return "error/error";  // 또는 적절한 fragment
}
```

### 2.5 결제·문의 화면 본인 여부 미검증 (보안/일관성)

**위치**: `ProductController.java`

- **GET `/product/purchase/{id}/payment`** (2264~2273):  
  `id`만으로 결제 폼을 열어주며, **로그인 사용자와 예약 소유자 일치 여부를 검사하지 않음.**  
  → 다른 사람 예약에 대한 결제 화면 노출 가능.

- **POST `/product/purchase/{id}/payment`** (2276~2333):  
  `principal`로 현재 사용자만 가져오고, **해당 예약이 현재 사용자 소유인지 검사하지 않음.**  
  → 다른 사람 예약에 대해 결제 처리될 수 있음.

- **GET `/product/purchase/{id}/inquiry`** (2344~2352):  
  마찬가지로 본인 예약 여부 검사 없음.

**권장**:  
`getPurchaseDetail`와 동일하게 “본인 예약인지” 검사 후, 아니면 403 또는 에러 페이지 반환.

- GET/POST payment, GET inquiry 모두:
  - `principal` 없으면 로그인 요청.
  - `purchaseProduct.getUser().getId().equals(currentUser.getId())` 확인.
  - GET payment에서도 `purchaseProduct`가 없으면 404 등 처리 (`.get()` 대신 `orElseThrow` 권장).

### 2.6 Optional.get() 사용 (예외/가독성)

**위치**: `ProductController.java`  
2267, 2283, 2284, 2347, 2361 등

**현재**:  
`findByIdWithAll(id).get()`, `findByEmail(...).get()` 등으로 Optional을 강제 해제.

**문제**:  
조회 실패 시 `NoSuchElementException`이 나며, 메시지가 불명확함.

**권장**:  
`orElseThrow(() -> new IllegalArgumentException("..."))` 등으로 바꿔, 404/400 등과 연계해 메시지와 응답을 명확히 함.

### 2.7 세션 복구 시 선택한 항공 조합과 불일치 (설계)

**위치**: `ProductController.java`  
`@PostMapping("/reservation")` — reservationData로 복구하는 부분 (2068~2084)

**현재**:  
`reservationData`에 `productId`, `departDate`만 있어도 복구할 때  
`productServiceCommon.calcSingleProduct(target, departDate)`만 호출함.  
이때 `target`에는 **특정 출국/귀국·좌석등급이 설정되어 있지 않으므로**,  
`calcSingleProduct` 내부에서 **최저가 항공편**으로 다시 계산됨.

**결과**:  
사용자가 달력/상세에서 선택한 “출국·귀국 시간 + 좌석등급”과 다른 조합으로 예약 페이지가 열릴 수 있음 (가격/편명 불일치).

**권장**:  
- `reservationData`(또는 `pendingReservation`)에  
  `departTime`, `returnTime`, `seatClassType`(또는 동일 의미의 식별자)를 포함해 저장하고,  
- 복구 시 `getProductDetail`과 동일하게 이 값들로 **동일 조합**을 찾아서 `calcSingleProduct`에 넘기거나,  
- 복구 시 “선택한 조합을 찾을 수 없으면 에러”로 두고, 상세에서 다시 선택하도록 유도.

### 2.9 비활성 상품 노출 (설계/정책)

**위치**: `ProductController.getProduct` (246~254)

**현재**:  
`isActive == false`인 상품을 막는 분기가 **주석 처리**되어 있어, 비활성 상품도 달력/상세에 노출됨.

**권장**:  
- 의도가 “일단 비활성도 조회 가능”이라면, 주석에 이유를 남기고, 필요 시 목록/검색에서는 비활성 제외.  
- 판매 중지 상품은 노출하지 않을 거라면, 주석 해제해 다시 활성만 허용하는 것이 일관적임.

## 3. 정상 동작으로 보이는 부분

- **노출**: 검색/목록에서 `calcSingleProduct` 호출, 가격·잔여석 계산 일관됨.
- **달력**: `findMultipleProductsForDate` + `hasHoldWithPoolSpace`로 “홀드 풀 있으면 availableSeats 없이 노출” 조건 반영됨.
- **상세**: 선택한 출발일·항공 조합으로 `calcSingleProduct` 호출, 세션에 `calcProduct` 저장.
- **예약 생성**: `ensurePackageSeatHold` 호출로 패키지 홀드 생성/사용.
- **결제 완료**: `findPackageSeatHold` + `allocatePackageSeatHold`로 홀드 allocated 증가.
- **탑승객**: `PassengerServiceCommon.updatePassengersFromDto`, `isPassengerInfoComplete` 반영.
- **내역**: `findByUserId`로 본인 예약 목록 조회, 환불 정보 매핑 후 리스트 노출.
- **결제 후 취소**: `requestRefund` → RefundService 등에서 `cancelPurchase` 호출 시 홀드/대기열 처리됨.
- **예약대기 → 순차 확정**: `cancelPurchase()` 내부에서, **같은 항공 조합(같은 PackageSeatHold: productId, departDate, outboundSeatClassId, returnSeatClassId)** 인 대기 예약만 후보로 필터한 뒤, `purchaseDate` 순으로 가용석이 허용하는 만큼 확정. 홀드가 있는 경우 해당 홀드의 `totalHeld - allocated`로 가용석 계산; 홀드가 없는 레거시는 (product, departDate) 단위로 기존처럼 처리.
